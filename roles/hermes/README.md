# hermes

Installs the [Hermes agent](https://hermes-agent.nousresearch.com/docs/) system wide and
runs its messaging gateway as a systemd service under a dedicated unprivileged `hermes`
user. All configuration goes through Hermes' **managed scope**, an administrator owned
directory the agent cannot rewrite.

The role deliberately ships **no configuration**: the managed files are created empty. Model,
API keys and messaging platform are supplied through the role variables below, or
interactively on the machine.

The installer pulls python, node, ripgrep and ffmpeg itself; the OS packages it expects to
find (`hermes_packages`, `git curl xz-utils build-essential` by default) are installed by
this role.

## Directory layout

| Path | Owner | Mode | Purpose |
|---|---|---|---|
| `/usr/local/lib/hermes-agent` | `root:root` | | code, installed by the root mode installer - the agent cannot modify it |
| `/usr/local/bin/hermes` | `root:root` | | launcher |
| `/etc/hermes` | `root:hermes` | `0750` | managed scope, read only for the service |
| `/etc/hermes/config.yaml` | `root:hermes` | `0640` | pinned settings, empty by default |
| `/etc/hermes/.env` | `root:hermes` | `0640` | managed environment, empty by default |
| `/var/lib/hermes` | `hermes:hermes` | `0700` | `HERMES_HOME`: the agent's own `config.yaml`, `.env`, `memories/`, `skills/`, `sessions/`, `logs/` |

Upstream documents the managed scope as `0755`/`0644`, world readable, and therefore warns
against putting secrets into the managed `.env`. This role tightens it to `0750`/`0640`
`root:hermes`: only the service can read it, so `hermes_managed_env_secrets` may hold API
keys while the agent still cannot change its own policy.

Managed values override the user scope **and** the shell environment for the keys they set,
at leaf level - pinning `model.default` leaves the rest of `model.*` user configurable.
Check what is in effect with:

```bash
sudo -u hermes HERMES_HOME=/var/lib/hermes hermes config
sudo -u hermes HERMES_HOME=/var/lib/hermes hermes doctor
```

## Service

The unit is generated by upstream (`hermes gateway install --system`, run once), so its
service definition stays compatible with the gateway. Identity, environment and hardening
come from the ansible owned drop-in
`/etc/systemd/system/hermes-gateway.service.d/10-ansible.conf`: `User`/`Group`,
`HERMES_HOME`, and `ProtectSystem=strict` with `ReadWritePaths` limited to the state
directory. The agent's own shell commands run inside that sandbox, so anything it should be
able to write to has to be listed in `hermes_service_read_write_paths`.

On a fresh machine the service is installed and enabled but **not started** - without a
model and a messaging platform the gateway would only crash loop. Configure the agent, then
run the role again:

```bash
sudo -u hermes HERMES_HOME=/var/lib/hermes hermes setup
```

Once a setting should no longer be changeable from a chat session, move it from the state
`config.yaml` into `hermes_managed_config`.

## Wiring in configuration other roles own

`hermes_managed_env` is the single wiring point for values that belong to another role: the
role itself knows no key names. The composing playbook includes the producing role with
`public: yes` - an `include_role` is `public: false` by default, so its variables would not
be visible afterwards - and maps them:

```yaml
      - name: Install signal-cli and its daemon
        ansible.builtin.include_role:
          name: andreasbehnke.ai_agent.signal_cli
          public: yes

      - name: Install the Hermes agent and its gateway
        ansible.builtin.include_role:
          name: andreasbehnke.ai_agent.hermes
        vars:
          hermes_managed_env:
            SIGNAL_HTTP_URL: "http://{{ signal_cli_http_host }}:{{ signal_cli_http_port }}"
            SIGNAL_ACCOUNT: "{{ signal_cli_account }}"
            SIGNAL_ALLOWED_USERS: "{{ hermes_signal_allowed_users | default([]) | join(',') }}"
```

Entries with an empty value are skipped, so optional keys can be mapped unconditionally.

Values that have no producing role - Hermes' own settings - are provided by the inventory,
named after the Hermes key they carry: `hermes_signal_allowed_users` for
`SIGNAL_ALLOWED_USERS`. Secrets never travel through this wiring, they go through
`hermes_managed_env_secrets` and stay in the password store until the template is rendered.

## Variables

| Variable | Default | Description |
|---|---|---|
| `hermes_user` / `hermes_group` | `hermes` | service identity, created as a system user without login shell |
| `hermes_service_name` | `hermes-gateway` | unit name generated by upstream |
| `hermes_packages` | `[git, curl, xz-utils, build-essential]` | OS packages the installer needs, installed by this role |
| `hermes_installer_url` | `https://hermes-agent.nousresearch.com/install.sh` | installer, run as root; there is no version pinning, use `hermes update` |
| `hermes_bin` | `/usr/local/bin/hermes` | launcher path the installer creates |
| `hermes_code_dir` | `/usr/local/lib/hermes-agent` | code path the installer creates |
| `hermes_config_dir` | `/etc/hermes` | managed scope; a different path is passed to the service as `HERMES_MANAGED_DIR` |
| `hermes_home` | `/var/lib/hermes` | `HERMES_HOME`, the writable state directory |
| `hermes_managed_config` | `{}` | keys pinned in the managed `config.yaml` |
| `hermes_managed_env` | `{}` | plain values of the managed `.env`, e.g. `SIGNAL_HTTP_URL` |
| `hermes_managed_env_secrets` | `{}` | secret values of the managed `.env`: variable name to password store entry |
| `hermes_service_hardening` | `true` | write the sandbox directives into the drop-in |
| `hermes_service_read_write_paths` | `[]` | additional writable paths for the agent |

## Example

```yaml
- hosts: agent
  become: yes
  tasks:
    - ansible.builtin.include_role:
        name: andreasbehnke.ai_agent.hermes
      vars:
        hermes_managed_config:
          model:
            provider: anthropic
            default: claude-sonnet-5
        hermes_managed_env:
          SIGNAL_HTTP_URL: "http://127.0.0.1:8080"
        hermes_managed_env_secrets:
          ANTHROPIC_API_KEY: "private/ai/anthropic"
```
